api AS conversation

A web API is a conversation in which there is a well known and fairly strict grammar. When that grammar is followed well many participants can engage in the conversation.

TESTS AS conversation

Tests are a conversation as well. They help discover the better grammar when creating things and validate the grammar as it grows. They become the authority on the grammar.

This is your brain on openstack

problems

  • Challenging to learn existing projects
  • No clear source of authority
  • Object-oriented, deeply subclassed, test-scenario-ized tests are very hard to read
  • Client code for sake of DRY hides HTTP, masking faults
  • Entrenched model of testing

G0als

  • Easily evaluate health and functionality of an API
  • Lower verbosity, raise transparency
  • Focus on HTTP (methods, headers, bodies)
  • Speed up writing, reading and running tests
  • Learning by reading and writing
  • TDD for APIs

demo!

  • Test any API with gabbi-run
  • Confirm status code and response headers
  • Check for strings in response bodies
  • Send any body data and request headers
  • Evaluate JSON responses with JSONPath
  • Reference the previous test to traverse APIs

features

  • Access environment variables
  • Poll asynchronous resources
  • xfail and skip for great good
  • Integrate with Python testrunners
  • Use nested fixtures
  • Provide a prefix for apps
  • Test across multiple services
  • Test without a server, use wsgi-intercept
